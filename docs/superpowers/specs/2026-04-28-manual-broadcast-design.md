# Ручная рассылка с выбором специализации — Дизайн

**Дата:** 2026-04-28
**Статус:** Утверждён

---

## Цель

Администратор может вручную запустить рассылку в любое время из `/admin`, выбрав одну из топ-5 специализаций по изменению цен, просмотрев превью и подтвердив отправку.

Автоматическая рассылка по понедельникам в 9:30 не меняется.

---

## Компоненты

### Новые файлы
- `handlers/broadcast.py` — весь сценарий ручной рассылки (список → превью → подтверждение)
- `tests/test_broadcast.py` — unit-тесты handler

### Изменения в существующих файлах

| Файл | Изменение |
|------|-----------|
| `db.py` | Новая функция `get_top_specializations(last_date, filters, limit=5)` — возвращает список вместо одной строки |
| `handlers/admin.py` | Кнопка «📨 Тест-рассылка» → «📨 Рассылка», `callback_data="admin_broadcast"` — входная точка в broadcast handler |
| `bot.py` | Подключить `broadcast.router` |

### Не меняется
`report.py`, `scheduler.py`, `config.py`

```
/admin → «📨 Рассылка»
    ↓
Бот: топ-5 специализаций inline-кнопками
  [⬆️ Хирургия +8.2%]
  [⬆️ Терапия +5.1%]
  [⬇️ Педиатрия -3.4%]
  [⬆️ Анестезиология +2.7%]
  [⬇️ Неврология -1.2%]
  [◀️ Назад]
    ↓
Администратор выбирает специализацию
    ↓
Бот: отправляет превью (то же сообщение, что увидят подписчики)
     + кнопки: [✅ Отправить всем] [❌ Отмена]
    ↓
  ✅ → рассылка всем активным пользователям
       «✅ Отправлено N пользователям»
  ❌ → «Рассылка отменена» + возврат в главное меню
```

## FSM

`BroadcastState.confirming` — сохраняет название выбранной специализации в `state.data["specialization"]` пока ждём подтверждения.

## Новая функция БД

```python
async def get_top_specializations(last_date, filters: list = None, limit: int = 5) -> list:
```

Идентична `get_top_specialization`, но:
- `pool.fetch(...)` вместо `pool.fetchrow(...)`
- `LIMIT $N` параметризован через `limit`

`since_date=None` при ручной рассылке — показываем все доступные данные.

## Формат кнопок специализаций

```
{emoji} {specialization} {sign}{pct}%
```

- `emoji` = `⬆️` если `net_change > 0`, иначе `⬇️`
- `pct` = `round(net_change / sum_old_price * 100, 1)` если `sum_old_price > 0`, иначе `0.0`
- `callback_data` = `"bcast_spec_{index}"` (индекс 0–4) — чтобы избежать лимита 64 байта Telegram на callback_data

Порядок: по убыванию `total_abs_change` (топ-1 сверху).

**Разрешение индекса в название:** когда администратор нажимает кнопку `bcast_spec_{index}`, handler повторно запрашивает `get_top_specializations(None, filters, limit=5)`, берёт элемент по индексу и сохраняет `spec["specialization"]` в `state.data["specialization"]` (`BroadcastState.confirming`). Повторный запрос допустим — данные меняются раз в неделю.

## Граничные случаи

| Ситуация | Поведение |
|----------|-----------|
| `price_monitoring` пустая / нет данных | «Нет данных для рассылки» + кнопка ◀️ Назад |
| Специализация выбрана, `build_report` → `None` | «Нет изменений цен по выбранной специализации» + кнопка ◀️ Назад |
| Нет активных пользователей | Превью показывается; после подтверждения: «Нет активных подписчиков» |
| Telegram-ошибка при рассылке | Логируем каждую ошибку, продолжаем следующему |
| Пользователь заблокировал бота | `deactivate_user()` — та же логика, что в `scheduler.py` |

## Что НЕ меняется

- `send_weekly_report` и расписание APScheduler — без изменений
- `last_processed_date` ручная рассылка не обновляет
- Фильтры `admin_filters` применяются и в ручной рассылке (те же, что и в авто)
